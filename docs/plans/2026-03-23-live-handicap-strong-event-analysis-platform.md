# 比赛强事件驱动的滚球让球盘分析台方案文档

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 建设一期面向滚球让球盘的强事件分析台，识别比赛中的伤停、红黄牌、长治疗和换人等强事件，联动盘口与时间变化，发现潜在短时赔率错位机会，并通过回放和模拟对冲验证策略有效性。

**Architecture:** 系统采用三段式结构：采集机负责视频和赔率流接入与标准化，推理机负责候选事件检测与 `Qwen-VL` 复核，前端分析台负责多场监控、单场详情、复盘和对冲模拟。V1 不接自动下单，只做机会发现、验证和展示。

**Tech Stack:** Python、FastAPI、Redis Streams、PostgreSQL、TimescaleDB、MinIO 或本地对象存储、React/Next.js、Qwen-VL

---

## 1. 项目目标

本项目一期的目标，不是直接做自动下注系统，而是建设一套面向滚球让球盘的实时分析台，用于在比赛进行中识别强事件，联动盘口与时间变化，发现潜在的短时赔率错位机会，并通过历史回放和模拟对冲验证策略有效性。

这里的“强事件”主要指对盘口影响通常较快、较明显、并且可以从视频中直接观察到的事件，包括：

- 球员重伤或疑似重伤倒地
- 红牌、黄牌、争议判罚
- 长时间治疗或比赛中断
- 换人、担架入场、裁判集中处理事件

一期系统的核心交付结果是：

- 实时监控多场比赛
- 自动识别强事件
- 判断事件可能影响哪一侧盘口
- 结合当前让球线、水位、比赛剩余时间给出机会评分
- 在历史比赛中回放并验证这些信号是否真的领先赔率变化

## 2. 业务逻辑理解

该系统的目标不是预测整场比赛最终谁赢，而是捕捉“事件发生”和“赔率重新定价”之间的短时间差。

典型场景一：

- A 队核心球员被严重犯规
- 市场短时间担心该球员退赛，A 队赔率可能先恶化
- 但如果犯规方因此红牌出场，后续盘口又可能重新修正，对 A 队形成利好
- 这段从事件发生到赔率完成重定价的窗口，就是系统要识别和提示的机会区间

典型场景二：

- 比赛进入 80 分钟后，落后方时间本已不多
- 如果此时发生长时间伤停、治疗或比赛中断，实际可进攻时间进一步减少
- 即使比分没变，盘口也会因“有效剩余时间减少”而快速重定价

因此系统需要建模的核心变量包括：

- 事件类型
- 事件影响对象
- 事件持续时长
- 当前盘口位置
- 当前水位水平
- 赔率变化速度
- 比赛剩余时间

## 3. 一期范围定义

### 3.1 纳入范围

一期只做以下内容：

- 滚球让球盘监控
- 强事件识别
- 视频与赔率时间轴对齐
- 实时机会评分与提示
- 历史复盘与模拟仓位验证
- 多比赛监控界面和单场详情页

### 3.2 不纳入范围

一期不做以下内容：

- 自动下注
- 实盘账户接入
- 浏览器自动点击或平台 API 执行
- 大小球、1X2、多盘口联动建模
- 攻势、阵型、控球压制等连续语义建模
- 盈利承诺或“绝对套利”承诺

一期的定位应明确为：发现和验证机会，而不是直接执行交易。

## 4. 总体架构

推荐采用三段式架构：

1. 采集机
2. 推理机
3. 前端分析台

### 4.1 采集机

负责接入现有的视频录制方案和实时赔率抓取方案，输出统一时间戳和标准化数据。

主要职责：

- 拉取或接收比赛视频流
- 切分短视频 clip 或关键帧
- 抽取电视画面中的比赛时间、比分、字幕、牌面信息
- 接入实时赔率 tick 数据
- 统一封装比赛标识和时间轴

### 4.2 推理机

部署本地 `Qwen-VL` 或相关视觉模型，负责对候选事件片段做语义复核和分类。

主要职责：

- 接收采集机送来的候选 clip
- 判断是否发生有效强事件
- 判断事件属于哪支球队
- 估计事件严重度和可能持续时间
- 给出对盘口影响方向的初步判断

### 4.3 前端分析台

提供多场比赛监控、单场详情、复盘和模拟面板。

主要职责：

- 接收后端实时推送的事件和机会提示
- 展示盘口、水位和事件时间线
- 支持回看事件 clip
- 支持输入模拟仓位并计算不同赛果下的盈亏

## 5. 核心系统模块

### 5.1 数据接入模块

这一层只做“真实世界事实采集”，不做复杂策略判断。

输入包括：

- 比赛视频流或录播文件
- 实时赔率 tick
- 比赛基础信息，如联赛、队名、开赛时间、比赛状态

输出包括：

- 视频帧或短 clip
- 比赛时钟
- 比分
- 标准化盘口数据
- 原始事件候选

### 5.2 时间对齐模块

这是整个项目最关键的基础模块之一。你目前已经有历史视频和赔率，但尚未完成对齐。若这一层不准，后面训练、回测和机会识别都会失真。

建议做法：

- 从电视画面 OCR 提取比赛时间
- 从赔率流中提取 tick 时间
- 建立视频时间与赔率时间的偏移量
- 允许人工校正
- 输出统一 `match_timeline`

目标是把历史样本的时间误差控制在 `±1 秒` 内。

### 5.3 候选事件生成模块

不建议一开始就让大模型全时段盯所有帧。更稳妥的方式是先用轻量规则或轻模型生成候选片段，再交给 `Qwen-VL` 复核。

候选触发条件可以包括：

- 球员倒地时间过长
- 比赛长时间未恢复
- 医疗队、担架、裁判围拢出现
- 出牌动作或红黄牌图标出现
- 换人牌出现

这一层的目标是尽量别漏掉明显事件，允许带一些噪声。

### 5.4 事件复核模块

候选片段送入视觉语言模型后，输出结构化事件对象。

建议最小输出字段：

- 事件类型
- 所属球队
- 严重度评分
- 置信度
- 预计持续时长
- 对盘口方向的影响判断

一期建议聚焦以下事件类型：

- `injury_serious`
- `injury_minor`
- `red_card_likely`
- `yellow_card_only`
- `treatment_long`
- `substitution_done`

### 5.5 机会评分模块

该模块不做下注决策，而是做机会排序与提示。

评分输入包括：

- 事件类型与严重度
- 当前让球线
- 当前水位
- 事件前后赔率变化斜率
- 比赛剩余时间
- 当前比分

评分输出包括：

- 是否值得关注
- 推荐关注哪一侧
- 有效观察窗口
- 当前参考赔率
- 达到何种赔率条件时适合做对冲模拟

### 5.6 对冲模拟模块

用于把业务思路落成可验证的数字模型。

输入：

- 第一次买入的方向、赔率、金额
- 第二次对冲的方向、赔率、金额

输出：

- 主队赢的盈亏
- 客队赢的盈亏
- 是否形成锁利
- 理论利润区间

这部分必须做成可复盘工具，因为它会直接决定后面是否值得进入实盘自动化。

### 5.7 前端监控模块

一期首页建议固定为“机会监控台”。

首页内容：

- 多场比赛列表
- 当前比分和比赛时间
- 当前让球盘和水位
- 最近事件标签
- 机会评分
- 风险颜色提示
- 事件发生后经过了多少秒

单场详情页内容：

- 视频 clip 或关键帧
- 盘口曲线
- 赔率变化曲线
- 事件时间轴
- 系统解释文本
- 仓位模拟器

复盘页内容：

- 按联赛、球队、事件类型筛选
- 查看某类事件发生后 10/30/60/90 秒赔率如何变化
- 输出命中率和误报样本

## 7. 技术方案建议

### 7.1 后端

建议使用：

- `Python + FastAPI`
- `Redis Streams` 作为实时事件流
- `PostgreSQL + TimescaleDB` 存储赔率时序和业务表
- `MinIO` 或本地对象存储保存 clip

原因：

- Python 适合处理视频、OCR、模型推理和策略验证
- FastAPI 适合快速搭建实时接口
- TimescaleDB 适合存储赔率 tick 这种时序数据
- Redis 适合做实时推送和解耦

### 7.2 模型侧

建议采用“轻量候选 + VLM 复核”的两段式方案。

原因：

- 全量视频直接送大模型，延迟高、成本高、稳定性差
- 候选过滤后送模型，可以把资源集中在关键片段
- 更利于后面扩展到多场比赛并行监控

### 7.3 前端

建议使用：

- `Next.js` 或 `React + Vite`
- 图表库用于盘口和赔率曲线
- WebSocket 实时推送事件卡片和机会提醒

## 8. 分阶段实施方案

### Phase 1：数据底座与时间对齐

目标：先把“能回放、能对齐、能看清楚事件与赔率关系”做起来。

交付物：

- 标准化比赛数据模型
- 历史视频与赔率对齐工具
- 时间偏移校准机制
- 基础数据库表结构
- 单场回放能力

成功标准：

- 选取一批历史比赛，视频与赔率时间误差控制在 `±1 秒`

### Phase 2：强事件识别原型

目标：先做出稳定的强事件检测，而不是追求覆盖所有场面。

交付物：

- 候选事件触发器
- `Qwen-VL` 复核服务
- 结构化事件输出
- 基本事件标注工具

成功标准：

- 在标注样本上，严重伤停、红牌场景、长治疗停顿能达到可用精度

### Phase 3：机会评分与模拟仓位

目标：让系统不只是“看到事件”，而是能判断这类事件是否值得关注。

交付物：

- 机会评分引擎
- 赔率变化窗口分析
- 对冲模拟器
- 单场解释页

成功标准：

- 在历史样本中统计出“事件出现后赔率按预期变化”的显著比例

### Phase 4：实时监控台

目标：支持同时盯多场比赛，并实时发现高价值机会。

交付物：

- 首页多场监控台
- 实时告警推送
- 单场详情页
- 历史复盘台

成功标准：

- 从事件确认到前端展示的总延迟不超过 2 秒
- 支持 4 到 8 场比赛同时监控

## 9. 测试与验证标准

### 9.1 时间对齐测试

验证视频时钟、赔率 tick、事件时刻是否一致。

目标：

- 历史样本对齐误差 `<= 1 秒`

### 9.2 事件识别测试

对不同事件类型分别统计 precision 和 recall。

建议一期目标：

- `precision >= 0.75`
- `recall >= 0.55`

这里应优先追求 precision，减少误报。

### 9.3 机会有效性测试

对每条机会提示，观察其后 `10/30/60/90 秒` 内赔率是否按预期变化。

统计指标：

- 机会数
- 有效机会数
- 无效机会数
- 不同事件类型命中率
- 不同比赛时间段命中率

### 9.4 模拟盈利测试

对历史比赛做离线回放，按照固定规则执行模拟仓位，统计理论结果。

输出指标：

- 达到目标对冲条件的比例
- 理论锁利分布
- 平均窗口长度
- 误报造成的潜在损失分布

## 10. 风险与限制

### 10.1 这不是严格意义上的无风险套利

该方案更接近事件驱动的短时交易，而不是数学上严格无风险套利。

原因包括：

- 赔率变化未必按预期发生
- 事件判断可能错误
- 平台延迟可能导致错过窗口
- 比赛可能在等待期间突然进球、红牌或逆转局势

因此系统应把重点放在：

- 提高发现速度
- 提高判断准确率
- 缩短提示延迟
- 做好历史验证

### 10.2 最大技术风险

一期最大的风险不是模型能力，而是时间对齐和样本标注质量。如果历史数据时间不准，模型就学不到真正领先赔率变化的信号。

### 10.3 最大业务风险

如果后面进入自动下单阶段，还会面临：

- 平台风控
- 下单速度
- 账户限制
- 合规问题
- 网络延迟和执行偏差

这些不应在一期同时解决。

## 11. 一期结论

综合当前条件：

- 有 Mac Studio M3 Ultra 96G
- 已有本地视觉模型能力
- 已有视频录制方案
- 已有赔率实时抓取方案

这个项目的一期是可做的，而且不应该一上来就做自动执行。最合理的路线是：

1. 先补时间对齐能力
2. 再做强事件检测
3. 再做机会评分和仓位模拟
4. 最后做多场监控台和复盘系统

如果这一套跑通，就能回答最核心的问题：

这类比赛中的强事件，是否真的能在赔率重定价前形成稳定、可重复、可验证的短时机会。
