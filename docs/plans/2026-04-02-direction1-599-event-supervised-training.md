# 方向1：599事件标签监督训练

> 日期：2026-04-02  
> 目标：利用155场已对齐的599 live text事件作为监督信号，自动构建大规模"视频帧→足球事件"训练集  
> 状态：实施中

---

## 一、核心思路

**之前的训练流程**（Phase A-F2）：
1. 人工筛选gold matches → 固定间隔采帧 → 35B VLM自动标注 → 人工审核 → LoRA训练
2. 瓶颈：人工标注慢、gold matches少（28场）、LoRA反而降低准确率

**新思路**：
1. 155场全场录像已有141,938条599事件，每条事件都有 `_video_pos_sec`（视频帧定位）
2. 599事件 = **天然的ground truth标签**（进球、射门、角球、犯规、黄牌、红牌、换人等）
3. 直接在事件时间点提取帧 → 599事件类型作为标签 → 无需人工标注
4. **数据量级提升**：从1,684帧 → 预计50,000+帧（30倍扩量）

## 三、帧采样策略

### 3.1 事件帧采样

对每个映射到P0-P2的599事件：

| 参数 | 值 | 说明 |
|---|---|---|
| 采样窗口 | `_video_pos_sec` ± 5秒 | 覆盖事件前后画面 |
| 采样间隔 | 每2秒1帧 | 窗口内取6帧：-4, -2, 0, +2, +4秒 |
| 中心帧标签 | 599事件类型 | confidence=1.0 |
| 边缘帧标签 | 同类型 | confidence=0.7（±2s）, 0.4（±4s） |

**预估产出**：
- P0事件（进球+红牌）：~800事件 × 6帧 = ~4,800帧
- P1事件（黄牌+射门+角球）：~15,000事件 × 6帧 = ~90,000帧
- P2事件（犯规+换人+射偏）：~30,000事件 × 6帧 = ~180,000帧
- **合计**：~275,000帧（远超需要，需做采样控制）

### 3.2 采样控制

为防止数据倾斜和存储爆炸：

| 事件类型 | 每场最多采样事件数 | 帧/事件 | 预估总帧数 |
|---|---|---|---|
| P0（进球/红牌） | 全部 | 6 | ~4,800 |
| P1（黄牌/射门/角球） | 20 | 4 | ~12,400 |
| P2（犯规/换人/射偏） | 10 | 2 | ~3,100 |
| **合计** | — | — | **~20,300** |

### 3.3 负样本采样

训练事件检测需要大量"无事件"负样本：

- 在两个事件之间的中间点采1帧，标签为 `none`
- 确保负样本数 ≈ 正样本数（1:1比例）
- 过滤：跳过 `_video_pos_sec` 距离任何事件 < 10秒的位置
- 预估：~20,000帧负样本

## 四、训练数据格式

### 4.1 与现有格式兼容

沿用现有 `frame_conversations.jsonl` 的对话格式，保持与 LiveObserver 推理一致：

```jsonl
{
  "conversations": [
    {"role": "system", "content": "你是直播画面分析助手。严格只输出纯 JSON，不要解释。"},
    {"role": "user", "content": "<observation_prompt>", "images": ["path/to/frame.jpg"]},
    {"role": "assistant", "content": "{\"scene_type\": \"live_play\", \"score_detected\": \"\", \"match_clock_detected\": \"\", \"scoreboard_visibility\": \"unknown\", \"replay_risk\": \"low\", \"tradeability\": \"tradeable\", \"event_candidates\": [{\"label\": \"goal\", \"confidence\": 1.0}], \"confidence\": 0.85, \"explanation_short\": \"进球画面\"}"}
  ],
  "metadata": {
    "source": "599_event_supervised",
    "match": "Bayern Munich W vs Manchester United W",
    "event_code": 1029,
    "event_text": "进球！主队破门得分！",
    "video_pos_sec": 1892.46,
    "sample_offset_sec": 0,
    "priority": "P0"
  }
}
```

### 4.2 assistant标签构建规则

599事件只提供 `event_candidates` 和部分 `score_detected` 信息。其余字段：

| 字段 | 来源 | 说明 |
|---|---|---|
| `scene_type` | 推断 | 事件帧→`live_play`，负样本→`live_play` |
| `score_detected` | 599 msgText解析 | 进球事件可提取比分，其余为空 |
| `match_clock_detected` | `_match_elapsed_sec` | 转换为 mm:ss 格式 |
| `scoreboard_visibility` | 留空`unknown` | 无法从599推断 |
| `replay_risk` | `low` | 事件发生时一般是实况 |
| `tradeability` | 推断 | 有事件→`tradeable`，负样本→`tradeable` |
| `event_candidates` | **599映射** | 核心标签，来自映射表 |
| `confidence` | 基于offset | 中心帧0.9，边缘帧递减 |
| `explanation_short` | 599 msgText | 直接用599文本 |

### 4.3 关键限制

- `score_detected` 和 `match_clock_detected` 多数情况**无法从599推断**（599只有比赛经过时间，不是屏幕上的时钟）
- `scoreboard_visibility` 完全无法推断
- **这意味着这批数据主要训练 `event_candidates` 检测能力，不用于训练OCR**
- OCR能力保留9B base原有水平（96.9%/98.8%），不在此方向覆盖

## 五、数据划分策略

**按比赛划分**（防止数据泄漏）：

| 集合 | 比赛数 | 说明 |
|---|---|---|
| 训练集 | 125场（~80%） | 随机选取 |
| 验证集 | 15场（~10%） | 不同联赛代表 |
| 测试集 | 15场（~10%） | 最终评估，不参与任何调参 |

**划分原则**：
1. 同一场比赛的所有帧只能出现在同一个集合
2. 验证集和测试集尽量覆盖不同联赛
3. P0事件（进球/红牌）在三个集合中按比例分配

## 六、评估指标

### 6.1 事件检测指标

| 指标 | 计算方式 | 目标 |
|---|---|---|
| **Event Detection Accuracy** | 预测的event_candidates.label == GT label | ≥ 80% |
| **Per-class Precision/Recall** | 每类事件单独计算 | 进球recall ≥ 90% |
| **False Positive Rate** | 负样本被误判为有事件 | ≤ 15% |
| **Mean Confidence Calibration** | 预测confidence与实际准确率的差 | |calibration error| ≤ 0.1 |

### 6.2 不退化指标（关键约束）

| 指标 | 基线（9B base） | 约束 |
|---|---|---|
| Score OCR准确率 | 96.9% | 不得下降 |
| Clock OCR准确率 | 98.8% | 不得下降 |
| JSON输出有效率 | 99.8% | 不得下降 |

### 6.3 评估方法

1. **自动评估**：在测试集15场上运行模型，自动计算上述指标
2. **OCR回归测试**：在原holdout 400帧上重新跑，确认不退化
3. **人工抽检**：随机抽100帧预测结果，人工判断合理性

## 八、角球套利核心分析

### 8.1 为什么是角球？

| 维度 | 角球 | 进球 | 红牌 |
|---|---|---|---|
| 每场频率 | 8-12次 | 2-3次 | 0-0.5次 |
| 盘口反应速度 | **有延迟** | 极快 | 快 |
| 套利窗口 | **存在** | 几乎没有 | 很小 |
| 可预判性 | **高**（进攻压到底线可预判） | 低 | 极低 |
| 数据量 | 1,862帧（充足） | 1,242帧 | 30帧 |

### 8.2 角球时序模型设计

角球不是一个瞬间事件，而是一个**时序过程**：

```
[角球前 -10s~-3s] → [球出底线 -3s~0s] → [角球判定 0s] → [摆放+罚球 +3s~+15s] → [争顶/解围 +15s~+25s]
     ↑                    ↑                   ↑
  预判窗口              检测窗口           599事件时间点
  (进攻方压到底线)       (球出底线画面)        (盘口开始变化)
```

- **预判**（最大价值）：在球出底线前就识别出"高概率角球"场景
- **快速检测**（保底价值）：在599通报角球的同时或更早检测到

### 8.3 角球前兆画面特征

| 时间 | 画面特征 | 检测难度 |
|---|---|---|
| -10s~-5s | 进攻方在对方半场压上，防守方收缩 | 高（需要理解队形） |
| -5s~-3s | 球在底线附近，进攻方尝试传中/突破 | 中（球在底线区域） |
| -3s~0s | 球出底线，裁判/边裁指向角球区 | 低（明显动作） |
| 0s~+3s | 球员跑向角球区摆放 | 极低（非常明显） |

### 8.4 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| **599事件时间偏移** | 角球帧可能已经是罚球画面而非出底线瞬间 | 用 -10s~+5s 多帧采样，标注时序阶段 |
| **回放画面混入** | 角球后可能穿插回放 | 角球回放较少（不像进球），影响小 |
| **预判误报** | 进攻方压到底线但没产生角球（界外球/门球） | 可接受的误报——宁可多看一次水位 |
| **3s/帧延迟** | 模型处理太慢，错过套利窗口 | Phase 2.4 优化：降分辨率/量化/批处理 |
| **视频质量参差** | 不同转播源画质不同 | 帧质量检查，过滤模糊/黑帧 |
