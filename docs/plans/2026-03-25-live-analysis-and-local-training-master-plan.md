## 1. 这份文档回答什么问题

这份文档不是“通用直播理解”计划，而是当前项目的**真实业务目标总计划**。

它回答的是：

1. 我们最终到底要做成什么
2. 为什么当前先做纯视觉 benchmark
3. 后面为什么一定要进入“事件 + 盘口数据”的联合判断
4. 什么时候才该进入训练/微调

## 2. 项目的真实目标

项目的最终目标不是：

- 让模型解说比赛
- 做一个通用足球理解模型
- 只看比分和比赛时间

项目的真正目标是：

**通过识别直播中的重定价事件，提前判断盘口即将如何变化，并利用事件前后赔率差完成低风险对冲套利。**

也就是说，系统要做的是：

1. 在直播中尽早发现“会引发盘口重定价”的场内事件
2. 结合实时盘口、时间、赔率变化理解这些事件的含义
3. 判断是否存在“先手买入 + 事件落地后对冲”的空间
4. 给出结构化的套利机会判断，而不是泛化的比赛描述

## 3. 典型业务场景

### 3.1 伤病 / 红牌风险联动

例如：

- 切尔西核心球员被重踩，疑似伤退
- 爱佛顿犯规球员可能黄牌/红牌

这类场景的重点不是“发生了犯规”，而是：

- 核心球员伤退会削弱切尔西
- 对手少一人又会削弱爱佛顿
- 市场对这两个因素的定价调整未必同步

套利价值就在于：

- 先在市场尚未完全重定价前买入第一腿
- 等赔率变化后，再买第二腿完成对冲

### 3.2 时间流失 / 医疗暂停

例如：

- 比赛进入 80 分钟后仍只有 0:0 或 1:0
- 落后方急需进球
- 这时一次伤停或治疗会消耗 1~2 分钟有效时间

这类场景中：

- 时间本身就是价值
- 剩余时间越少，落后方想打穿盘口越难
- 赔率与让球会随时间流失继续重定价

这里的目标不是识别“有人倒地”，而是判断：

- 这次暂停是否足够长
- 是否会显著改变市场对剩余时间价值的估计

## 4. 当前为什么先做纯视觉 benchmark

当前纯视觉 benchmark 不是为了直接给出套利建议，而是验证：

- 模型有没有资格当“场内事件观察员”

当前先做的基础能力包括：

- 比分识别
- 比赛时间识别
- 镜头状态识别
- 强事件候选识别

原因很简单：

- 如果模型连场上的强事件都看不稳
- 后面就不可能稳地和盘口数据联动

所以当前阶段只是：

- **事件观察能力验证**

还不是：

- **套利决策模型**

## 5. 系统的最终分层

整个系统后续应明确分成四层。

### Layer 1. 场内事件识别层

负责看视频，识别：

- 伤病事件
- 红牌风险
- 医疗暂停
- 时间流失
- 场面压制
- 回放 / 庆祝 / 争议判罚 / VAR 等强事件候选

### Layer 2. 盘口状态理解层

负责理解：

- 当前让球线
- 当前赔率/水位
- 比赛还剩多少时间
- 当前盘口是否已经开始重定价
- 重定价方向和速度

### Layer 3. 事件-盘口联动判断层

负责判断：

- 这个事件是否足以引发盘口变化
- 盘口大概率往哪边变
- 变化幅度可能有多大
- 是否存在先手买入的价值

### Layer 4. 风险与执行层

负责输出：

- 是否值得出手
- 第一腿买哪边
- 第二腿何时对冲
- 是否具备锁利空间
- 如果判断错误，风险暴露有多大

## 6. 硬边界

当前项目必须始终遵守以下边界：

- 只使用“视频可播 + 数据已绑定”的比赛
- 无数据绑定比赛：
  - 不录
  - 不入库
  - 不评测
  - 不训练
- 当前录制系统是唯一合法样本入口
- 先做分析、评测和复盘，不做自动下注执行

## 7. 完整阶段路线图

### Phase 0. 数据基础层固化

目标：

- 继续稳定 `recordings + live_dashboard`
- 保证只录合法比赛
- 保证时间线、同步 viewer、分析成片稳定产出

通过标准：

- 连续多轮录制中，合法比赛产物稳定生成
- 无数据绑定比赛不再污染样本源
- 巡检、补位、结束关闭窗口、飞书回报稳定

### Phase 1. 场内强事件识别模型

目标：

- 训练/验证“场内强事件识别模型”
- 不是训练套利模型

重点任务：

- 伤病事件
- 红牌风险
- 医疗暂停
- 时间流失
- 场面压制

这一阶段的输出不是“下注建议”，而是结构化事件判断。

### Phase 2. 事件 + 盘口数据联合评测

目标：

- 让系统从“看见事件”升级为“理解事件对盘口的影响”

做法：

- 给模型看比赛片段
- 同时给它当时那一段盘口数据
- 让它判断：
  - 这个事件是否会引起盘口重定价
  - 向哪边变
  - 变化是否值得先手买入

### Phase 3. 套利机会评分器

目标：

- 把“事件 + 盘口”判断变成可执行的结构化评分

输出不是解说，而是：

- 是否值得出手
- 第一腿买哪边
- 预期赔率变化方向
- 预期第二腿时机
- 是否具备对冲空间

### Phase 4. 训练 / 微调

目标：

- 在前面三阶段稳定后，才决定是否做 LoRA / SFT

训练对象不是泛化“看比赛模型”，而是更窄的：

- **直播事件 -> 盘口影响 -> 套利机会判断模型**

进入训练前必须满足：

- 样本量足够
- 标签质量稳定
- 知识库/规则/缓存路线已经到瓶颈
- benchmark 能证明训练有必要

## 8. 当前阶段真正该做什么

当前最重要的不是：

- 直接训练套利模型
- 直接让模型输出买哪边

而是：

1. 建立强事件标签体系
2. 建立事件 + 盘口联合评测机制
3. 建立套利机会评分 schema
4. 继续做纯视觉与联合输入 benchmark

## 9. 当前不该做什么

当前不应该：

- 把当前工作误判成“做通用体育视觉模型”
- 把无数据比赛混入训练链
- 在没有事件-盘口联合评测前直接训练套利模型
- 在没有稳定评测集前直接开始微调

## 10. 下一步交付物

按当前计划，紧接着要补齐三类正式资产：

1. 强事件标签体系
2. 事件 + 盘口联合评测标准
3. 套利机会评分 schema

只有这三块清楚后，后续 benchmark、知识库建设和训练准入才会真正对齐业务目标。
